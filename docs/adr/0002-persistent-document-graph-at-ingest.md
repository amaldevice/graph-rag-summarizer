# Persist a document-scoped graph artifact during Ingest Runs

**Status:** Accepted
**Date:** 2026-07-13

## Context

Ingest Runs need one reusable, document-scoped graph-artifact lifecycle. The current Full-Pipeline Run still builds graph state after query retrieval, which repeats entity and relation extraction for every query and keeps persistence coupled to query-time execution.

This ADR owns only the persistent graph artifact, storage path, manifest lifecycle, fencing, status reporting, replacement safety, and backfill behavior. Adaptive topology and query-time context allocation are owned by stacked follow-on PRs #64/#65 and planning issues #60/#61.

This is an internal, optional stage of Ingest. The external launcher/operator contract, Query-Only behavior, and downstream behavior remain unchanged.

## Decision

Add an internal optional graph-artifact stage to Ingest Runs without changing the external launcher or operator contract.

1. Ingest reserves and claims the pending manifest entry and version under the manifest fence before any Qdrant or artifact work.
2. Ingest writes vectors and payloads to Qdrant as before.
3. Ingest may build or refresh the document graph artifact from the same full-document chunks and embeddings.
4. Ingest writes the versioned artifact to `graphs/{collection}/{document_id}/v{version}/graph.json.gz`.
5. Ingest writes the active manifest to `graphs/{collection}/manifest.json` and atomically activates a validated artifact by updating the manifest entry for that document only after validation succeeds.
6. Query-Only and downstream behavior stay unchanged; any consumer that reads the manifest must use the ordered tombstone preflight below before compatibility fallback when no usable artifact exists.

The collection manifest is authoritative and stores one entry per document. Readers resolve the entry for a given `document_id` from `graphs/{collection}/manifest.json`, then follow that entry's active pointer.

The manifest includes collection-level concurrency metadata:

- `manifest_revision` — monotonically increasing manifest revision for writers.
- `tombstone_epoch` — monotonically increasing collection epoch incremented in the same manifest CAS whenever a tombstone is created, cleared, or changed.
- `collection_fence_token` — collection-wide fence that invalidates all document fences while a replace-collection or collection-setup operation owns it.
- `collection_operation_id` / `collection_attempt_id` — current collection-wide owner and attempt; null outside collection operations.
- `manifest_backend` — immutable canonical backend-authority descriptor `{kind, namespace}` for this collection's manifest, lock records, and artifacts, where `namespace` binds the account plus bucket/container and endpoint namespace; a backend switch requires an explicit migration, never implicit fallback.
- `tombstone_set_digest` — committed digest of the complete tombstone proof set; the empty set is the digest of JCS `[]`.
- `pending_tombstone_set_digest` — candidate tombstone-set digest, or the literal `"pending"` marker while a collection tombstone epoch is being proved; readers fail closed while it is non-null.

`manifest_revision` is persisted in JSON and increments on every successful manifest CAS. The object-store ETag is out-of-band storage metadata, not persisted JSON; readers and writers obtain the manifest bytes plus ETag from storage. Existing manifests publish with CAS/`If-Match`. When the manifest does not yet exist, writers create it with `If-None-Match:*`; if ETag support is unavailable they must use a collection-scoped fenced lock/lease around the read-modify-write sequence instead of a plain lock. Lock acquisition returns a monotonically increasing fencing token; every manifest write carries that token; the storage adapter rejects any write whose token is lower than the latest accepted token; expired holders cannot write. If the selected backend cannot provide conditional writes or enforce fencing tokens, the manifest update must fail closed and abort rather than proceed with an unsafe plain lock. Stale writers must retry or abort; they must never overwrite a newer entry. The normal `If-Match` / `If-None-Match` path remains unchanged. `ManifestStore.read_snapshot()` obtains read authorization under the collection ETag/fence/coordinator, captures the exact manifest bytes, ETag when available, persisted `manifest_revision`, the accepted fallback fence token when ETag is unavailable, and `SHA-256(exact manifest bytes)` as one snapshot tuple, then releases that read authorization only after the tuple is complete. Readers must revalidate that same tuple, never pair a current lock token with an older manifest read; revalidation obtains a fresh authorization rather than reusing a released one.

If ETag-based CAS is unavailable, the fallback must be implementable with a durable per-collection lock record stored on the same object-storage backend at `locks/{collection}.json`, or an explicitly configured durable coordinator with the same guarantees. The lock record must be updated with conditional `If-Match` / `If-None-Match` semantics to issue monotonically increasing fencing tokens. Manifest write/CAS requests must carry the token as metadata, and the adapter must reject lower tokens atomically. If the backend or coordinator cannot provide both durable token issuance and token-enforced manifest writes, the operation must abort and fail closed. This fallback supplements, but does not replace, the normal ETag path.
If the object store cannot conditionally update even the lock record, a configured external coordinator is mandatory; a best-effort in-process lock is never an allowed fallback. The coordinator's accepted token must be returned by `ManifestStore.read_snapshot()` together with the exact bytes it authorized.

The durable claim that reserves a pending entry also fences the data plane for the same document and collection. Manifest CAS always uses the collection-level ETag plus `manifest_revision`, or the collection lock token on the fallback path. `manifest_revision` is only the manifest CAS/version precondition; `collection_fence_token` is the sole collection-scoped Qdrant data-plane fence. Document-scoped append/replace Qdrant mutations additionally use the entry's `document_fence_token`, issued monotonically by the document claim CAS, and capture the current `collection_fence_token`; they do not compare the document token to an unrelated collection revision. A replace-collection or collection-setup operation first claims a new `collection_fence_token`, marks the collection operation in the manifest, waits for or cancels prior document claims, and rejects every document operation whose captured collection token is no longer current. The collection fence therefore invalidates all document fences for the duration of the collection operation. To isolate delayed stale writes, graph-stage chunk point IDs are attempt-scoped as `UUID5(UUID_NAMESPACE_URL, "ingest-point:{collection}:{document_id}:{document_generation}:{attempt_id}:{chunk_uid}")`; a pending build uses `pending_attempt_id`, while published points and the document control point use the promoted `document_attempt_id`. The prior stable point id is retained only as `source_point_id` metadata during migration. Deletes use generation + attempt filters and never delete a newer generation. The claim consists of `operation_id`, `document_generation`, `source_fingerprint`, `pending_attempt_id`, `build_attempt_id`, `pending_version`, and the applicable scope-specific token(s). Every Qdrant data-plane or control-plane mutation—including collection creation, schema/index creation, control-point writes, upserts, updates, deletes, and cleanup—must go through the collection/document `FenceCoordinator`. The coordinator serializes the operation, validates scope + token + attempt + generation immediately before the call, and rejects stale or expired callers; no direct Qdrant mutation may bypass it. Qdrant payloads and control records carry the generation, attempt identity, and applicable scope-specific fence token. If the backend cannot issue or enforce this hierarchy, the operation fails closed.

Each manifest entry contains:

- `document_id` — document identifier.
- `document_generation` — positive integer generation persisted in the Qdrant document payload and manifest entry.
- `source_fingerprint` — fingerprint for the source input that produced the bound graph.
- `artifact_digest` — SHA-256 digest of the canonical graph JSON bytes; it may be null when no active artifact exists.
- `pending_artifact_digest` — candidate artifact digest for the pending version; null until the candidate bytes are built and validated.
- `operation_id` — durable ingest operation identifier for the artifact build or backfill.
- `pending_operation_id` — current owning operation for a pending manifest entry; null when no operation is pending.
- `pending_generation` — generation claimed by the current pending operation; null when no operation is pending.
- `pending_source_fingerprint` — source fingerprint claimed by the current pending operation; null when no operation is pending.
- `pending_backend` — canonical backend-authority descriptor selected for the pending artifact; must equal `manifest_backend` byte-for-byte; backend migration is outside this ADR and requires a separate migration decision.
- `pending_artifact_key` — exact pending artifact key; null when no operation is pending.
- `pending_attempt_id` — unique attempt identifier for the current pending build attempt; null when no operation is pending.
- `document_attempt_id` — the currently published Qdrant attempt selector for this document generation; null until a first publish. During a pending replacement it remains the prior published attempt, and the final publish CAS promotes the pending attempt into this field.
- `next_version` — the next free artifact version number, not a reserved version; it is advanced atomically with the reservation and never moves backward.
- `pending_version` — artifact version reserved by the current pending operation; null when no operation is pending.
- `active_version` — the published artifact version; null when there is no active artifact.
- `active_artifact_key` — the published artifact pointer; null when there is no active artifact.
- `build_attempt_id` — unique attempt identifier for the final CAS/publish attempt; null unless an operation is in flight or recently finalized.
- `status` — the current artifact state.
- `backend` — the canonical backend-authority descriptor that owns the artifact bytes; it may be null when no active artifact exists and must equal `manifest_backend` whenever non-null.
- `previous_pointer` — the prior active pointer for this document; audit/recovery only and null when no prior active artifact exists. It is a structured recovery object containing `artifact_key`, `version`, `backend`, `artifact_digest`, and `document_generation`.
- `updated_at` — last manifest update timestamp.
- `failure_reason` — optional diagnostic metadata for failures and rebuilds.
- `tombstone_operation_id` — durable replace-collection operation identifier for a tombstoned entry; null unless the entry was tombstoned.
- `tombstone_attempt_id` — durable replace-collection attempt identifier for a tombstoned entry; retained while tombstoned and cleared only by a fresh replace-document generation/attempt/fence.
- `tombstone_fence_token` — collection fence captured by the tombstone attempt and retained for exact control-point validation.
- `document_fence_token` — monotonically increasing token for document-scoped data-plane operations; collection-scoped replace operations use the collection fence instead.
- `version_ledger` — embedded monotonic version history atomically CAS-updated with this entry; each item is `{version, artifact_key, artifact_digest, backend_authority, source_fingerprint, operation_id, document_generation, attempt_id, fence_token, fence_scope, state, reserved_at, updated_at}`.

The active Qdrant selector is the pair `(document_generation, document_attempt_id)`. A pending attempt may write and validate points under `pending_attempt_id`, but it is never query-visible until the final manifest CAS verifies the artifact/control proof and atomically promotes that value to `document_attempt_id`. A superseded attempt therefore cannot remain visible merely because it shares a generation number.


The artifact blob metadata may carry `document_generation`, `source_fingerprint`, `artifact_digest`, `operation_id`, `pending_operation_id`, `pending_generation`, and `pending_source_fingerprint` as provenance, but those fields never grant ownership. `operation_id` remains deterministic for idempotent operation identity; `pending_attempt_id` and `build_attempt_id` are unique per build attempt and are generated per attempt so a superseding retry gets a new attempt id while an interrupted, non-superseded resume reuses the same attempt id. Writers, resumers, and readers must gzip-decompress `graph.json.gz`, validate the canonical uncompressed bytes, require the active `backend` and any non-null `previous_pointer.backend` to equal `manifest_backend`, and require the SHA-256 of those bytes to equal the active manifest entry's `artifact_digest`, or, while pending, `pending_artifact_digest`, plus blob metadata `artifact_digest` and, when validating recovery bytes, `previous_pointer.artifact_digest`; any backend or digest mismatch means the artifact is not available and the operation fails closed. Qdrant persists `document_generation`, `document_attempt_id`, `source_fingerprint`, and the fence-bearing control metadata; resume and publish authorization come from the current manifest CAS fields plus the caller's inputs, and the artifact metadata plus current Qdrant generation/source/attempt values are validation-only checks. The canonical artifact bytes are deterministic: serialize the graph JSON with RFC 8785 JSON Canonicalization Scheme rules as UTF-8, with deterministic key ordering and number/string normalization and no NaN/Infinity, then gzip the canonical bytes only for transport with fixed compression level, `mtime=0`, and an empty filename. `artifact_digest` is the SHA-256 of the canonical uncompressed graph JSON bytes, and state-artifact metadata repeats that exact digest. PR A must include a fixture test proving identical canonical graph input yields identical bytes and digest, and differing graph input changes the bytes and digest.

Before a document publish, the writer creates a deterministic UUID5 document control point named `graph-control:document:{collection}:{document_id}:{document_generation}:{pending_attempt_id}` with exact fields `graph_control_point=document`, `document_id`, `document_complete=true`, `document_generation`, `source_fingerprint`, `document_attempt_id=pending_attempt_id`, `document_fence_token`, `point_count`, and `point_set_digest`. A proof item is exactly `{"point_id": str(point.id), "payload": point.payload}`; payload is the full stored JSON payload, vectors are excluded, items are sorted by UTF-8 `point_id`, duplicate IDs fail closed, and the digest is SHA-256 of the RFC 8785/JCS UTF-8 bytes of the item array. The finalizer fully paginates the pending attempt's Qdrant points, excludes control points, and requires every point to carry the claimed generation, `document_attempt_id == pending_attempt_id`, source fingerprint, and document fence; it must match the control-point count/digest. Mixed generations or attempts, partial pagination, missing control proof, or any mismatch fails closed. The final publish CAS promotes `pending_attempt_id` into the manifest's `document_attempt_id`; the control point already carries that same promoted value. This control point is the Qdrant proof that independently batched writes represent one document generation and attempt. Tombstone-set digests use the same proof-item format and include the point-id/payload pair for every tombstone control point.
The document control-point JSON field names are exact: `graph_control_point=document`, `document_id`, `document_complete=true`, `document_generation`, `source_fingerprint`, `document_attempt_id`, `document_fence_token`, `point_count`, and `point_set_digest`; generic “attempt” or “fence” wording means these named fields.

`source_fingerprint` is `SHA-256(RFC8785/JCS_UTF8({"document_id": document_id, "chunks": [{"chunk_uid": ..., "chunk_id": ..., "text": ..., "hierarchy": ..., "layout": ..., "page_no": ..., "image_urls": ...}, ...]}))`, where chunks are ordered by numeric `chunk_id` then UTF-8 `chunk_uid`, `image_url` is normalized to a one-element `image_urls` list before hashing, missing optional fields are normalized to null/empty lists, and vectors, timestamps, and transient provider metadata are excluded. Any change to this canonical source descriptor creates a new fingerprint and generation.

The pointer/status contract is explicit:

| Status | `active_version` / `active_artifact_key` | Reader authority | Generation / fingerprint match | Reader behavior |
| --- | --- | --- | --- | --- |
| `available` | Non-null | Authoritative | Must match current `document_generation` and `source_fingerprint` | Use the manifest entry |
| `partial` | Non-null | Not authoritative | May match current `document_generation` and `source_fingerprint`, but is visible only for diagnostics / optional inspection | Compatibility fallback |
| `pending` | Prior active pointer when one exists; null on first-document pending/unavailable | Not authoritative | Not yet published against current Qdrant data | Compatibility fallback |
| `stale` | Prior active pointer when one exists; null on first-document pending/unavailable | Not authoritative | Current Qdrant data no longer matches the in-flight or superseded graph | Compatibility fallback |
| `unavailable` | Null on first-document pending/unavailable; otherwise prior active pointer stays as recovery-only bytes when one exists | Not authoritative | No usable active artifact exists | Compatibility fallback |
| `tombstoned` | Null | Deny state | Non-discoverable tombstone for a document omitted by replace-collection; active fields are cleared only after the newest recoverable pointer is preserved in `previous_pointer` | Skip document entirely; never use compatibility fallback; require Qdrant tombstone deny control point |

When `active_artifact_key` is null, the active-entry `backend` and `artifact_digest` are null as well. For `tombstoned`, active fields are null, pending fields are cleared, `document_generation`, `source_fingerprint`, `next_version`, and `tombstone_attempt_id` are retained, and `failure_reason` carries the tombstone reason. A tombstoned entry is intentionally non-discoverable and never enters compatibility fallback. Replace-collection writes a durable Qdrant-side tombstone deny control point with valid Qdrant point id `UUID5(UUID_NAMESPACE_URL, "graph-control:tombstone:{collection}:{document_id}")`; the raw `SHA-256(collection + NUL + document_id)` remains an audit payload field. The zero-vector control point carries `graph_control_point=tombstone`, `document_id`, `graph_tombstoned=true`, `tombstone_epoch`, tombstone `document_generation`, `tombstone_operation_id`, `tombstone_attempt_id`, fence token, and `tombstone_complete=true`. The writer must wait for the write, read the exact point back, and verify the stored payload before considering the tombstone durable; if read-back is missing, mismatched, or incomplete, retry under the same fence or fail closed.

Reader preflight is ordered and mandatory: (1) read the collection manifest snapshot, `tombstone_epoch`, and `collection_fence_token`; (2) completely enumerate and validate all Qdrant tombstone control points, including their UUID, document id, epoch, generation, operation, attempt, fence, and completion proof; (3) require the exact predicate `point_id == UUID5(UUID_NAMESPACE_URL, "graph-control:tombstone:{collection}:{document_id}")`, `graph_control_point == tombstone`, `graph_tombstoned == true`, `control.tombstone_epoch == manifest.tombstone_epoch`, `control.tombstone_fence_token == entry.tombstone_fence_token`, `control.collection_fence_token == manifest.collection_fence_token`, and control document/generation/operation/attempt fields equal the corresponding tombstoned manifest entry; (4) compute a deterministic SHA-256 over the RFC 8785/JCS UTF-8 canonical JSON of the point-id-and-payload-sorted control-point pairs; (5) exclude every denied document id before graph or vector compatibility fallback, including a Qdrant filter that excludes `graph_control_point` records from ordinary chunk search. A Qdrant deny record wins for exclusion when it disagrees with the manifest. A manifest tombstone without a matching complete deny record, an unavailable manifest, an incomplete control-point enumeration, or any mismatch/uncertainty fails closed before retrieval. The manifest remains authoritative for normal artifact state; Qdrant is the deny-safety authority only. The deny record may be cleared only by a fresh replace-document generation/attempt/fence CAS that increments `tombstone_epoch`. If the entry was pending and still had a current active pointer, promote that exact pointer into `previous_pointer` before clearing active fields; preserve the newest recovery pointer if one already exists.

The tombstone control-point JSON field names are exact: `graph_control_point`, `document_id`, `graph_tombstoned`, `tombstone_epoch`, `document_generation`, `tombstone_operation_id`, `tombstone_attempt_id`, `tombstone_fence_token`, `collection_fence_token`, and `tombstone_complete`. The generic fence/attempt wording above means these named fields, never an untyped substitute. The committed `tombstone_set_digest` is the expected digest for all structurally valid tombstone control points. A structurally valid orphan/stale deny conservatively wins for exclusion even without a current tombstoned entry; malformed or incomplete proof fails closed. While a new epoch is being published, the writer rewrites every retained tombstone control point to the new epoch/fence before computing and CAS-committing `tombstone_set_digest`; the manifest stays fail-closed until that digest is committed.

Digest and deny validation are separate paths: a control point for a currently tombstoned manifest entry must satisfy the exact epoch/generation/operation/attempt/fence predicate; a structurally valid orphan or stale control point is still a deny and is excluded conservatively, but is not treated as a matching manifest proof. The reader accepts the tombstone set only when the computed digest is exactly equal to `manifest.tombstone_set_digest`; any non-null `pending_tombstone_set_digest`, absent expected digest, or equality failure fails closed.

Readers retain a manifest snapshot token from preflight: the ETag when available, otherwise the exact `manifest_revision` + fallback fence token + manifest-byte digest returned by `ManifestStore.read_snapshot()`. They also retain `tombstone_epoch` and the control-point set digest, perform graph/vector selection, then re-read the manifest and completely re-enumerate the control points before returning results. If the snapshot token, epoch, or set digest changed, the reader discards the selection and retries from preflight; if the fallback token cannot be obtained or revalidation remains inconsistent, it fails closed. Replace-collection cleanup must exclude all `graph_control_point` records from deletion and legacy classification.


`pending` is written only after the pending manifest entry and version have been claimed under the manifest fence, and only then may Qdrant work begin. The pending entry keeps the current prior pointer in `active_version`, `active_artifact_key`, `backend`, and `artifact_digest` for recovery/readers while remaining non-authoritative; first-document pending/unavailable has a null pointer. While `pending`, readers must fall back to the compatibility path only after tombstone preflight. A pending manifest update records ownership only in the manifest by CASing `pending_operation_id`, `pending_generation`, `pending_source_fingerprint`, `pending_attempt_id`, `pending_version`, `pending_artifact_digest`, `document_fence_token`, and the captured `collection_fence_token`; that scope-specific fence is rechecked before every Qdrant batch. Collection-wide operations use the collection fence instead. All publish/failure CAS predicates also include `pending_backend` and `pending_artifact_key`. The finalizer must separately re-read the current Qdrant `document_generation` and `source_fingerprint` before publish or failure handling, and that reread is validation-only, not ownership. The same build attempt keeps the same `pending_attempt_id` / `build_attempt_id` across an interrupted but not superseded resume; a superseding retry must mint a new attempt id, and the old attempt id cannot finalize or continue Qdrant work. On every final publish or failure transition, the CAS predicate must include `operation_id`, `document_generation`, `source_fingerprint`, `pending_operation_id`, `pending_generation`, `pending_source_fingerprint`, `pending_version`, `pending_artifact_digest`, `pending_attempt_id`, `build_attempt_id`, `document_fence_token`, `collection_fence_token`, and the current manifest ETag or fencing token; a superseded old attempt cannot publish or mark failure even if every other field still matches. This publish/failure CAS is separate from the stale-transition CAS used after a Qdrant mismatch. After a mismatch, the stale-transition CAS compares `operation_id`, `pending_attempt_id`, `build_attempt_id`, `document_generation`, `source_fingerprint`, `pending_version`, `pending_artifact_digest`, `document_fence_token`, `collection_fence_token`, and the current manifest ETag or fencing token; if that CAS fails, do nothing. On successful publish, the new pointer replaces the retained recovery pointer and the replaced pointer becomes `previous_pointer`. If a newer operation supersedes the pending one, the older reservation/version is permanently burned, the older operation aborts, and it cannot resume or change status. Tombstoned entries are never rediscovered by stale work: resumers and finalizers must present the current manifest fencing fields for the same operation, and any future reintroduction of a tombstoned `document_id` must use a fresh replace-document generation plus CAS; append with the same `document_id` remains rejected.

Optional graph failure is a terminal, fenced transition rather than an abandoned `pending` state. If graph construction, canonicalization, artifact upload, Qdrant proof, or final validation is skipped or fails, PR A performs one CAS with the same operation/generation/source/attempt/fence predicates: it marks the pending ledger item `failed`, clears every `pending_*` field and `build_attempt_id`, records a bounded `failure_reason`, and sets `status=partial` when a prior validated pointer remains usable or `status=unavailable` when no usable pointer exists. It preserves the prior pointer only as recovery metadata and never promotes an unproved artifact. Scoped cleanup may remove points owned by the failed pending attempt; if cleanup is uncertain, the manifest remains non-authoritative for that attempt and the next retry must use a fresh attempt. Vector ingest remains successful unless its own vector contract fails. A later retry reserves a fresh version/attempt under CAS and cannot reuse a failed reservation.

PR A implements the following safety seams, which are intentionally not present in this documentation-only PR: `ManifestSnapshot` is the exact tuple returned by `ManifestStore.read_snapshot()`; `ManifestStore.cas(expected_snapshot, update, scope_token)` is the only manifest mutation; `ManifestStore.revalidate(snapshot)` obtains a fresh authorization and compares the full tuple; `FenceCoordinator.mutate(scope, fence_token, operation_id, generation, attempt_id, mutation)` is the only Qdrant mutation boundary; and `ActiveSelectorFilter` builds an authorized OR-filter over the collection's current `(document_id, document_generation, document_attempt_id)` entries while always excluding control points and denied ids. PR A acceptance tests must exercise superseded attempts, stale deletes, failed optional graph transitions, pagination/proof gaps, tombstone epoch/fence/digest mismatch, snapshot retry, backend-namespace mismatch, and version burns.

`document_generation` is always a positive integer. All chunks for one ingest share the same generation. A new append starts at generation `1`; a replace increments the previous generation. Backfill with a valid `document_id` and no existing generation derives generation `1` from one consistent source fingerprint and writes it once; inconsistent or malformed payloads are unavailable/rebuild-required and are never guessed. `available` must match the current `document_generation` and `source_fingerprint`; `partial`, `pending`, `stale`, and `unavailable` use compatibility fallback only after the mandatory tombstone preflight proves the document is not denied; `tombstoned` is always skipped. Legacy-point detection must scroll with offset until Qdrant returns `None`, validate every returned point, exclude any point with `graph_control_point` from legacy classification, and fail closed on any page, offset, or filter uncertainty. If Qdrant has already been replaced but graph activation fails, the old pointer may remain in active fields as recovery bytes until a newer publish, but it is never reader-authoritative; after publish the old bytes are retained in `previous_pointer`.

Version allocation is immutable and ledgered: `version_ledger` is embedded in the collection manifest and atomically CAS-updated with the document entry; object listing is never authoritative. Each ledger item has `version`, `artifact_key`, `artifact_digest`, `backend_authority`, `source_fingerprint`, `operation_id`, `document_generation`, `attempt_id`, `fence_token`, `fence_scope`, `state`, `reserved_at`, and `updated_at`. Every append reserves `v1`; replacement reserves `pending_version = next_version`, then increments `next_version` to the next free integer in the same CAS before Qdrant work; `next_version` is never a reserved value and no version is reused. Ledger state is per-version: `reserved` while pending, `published_available` or `published_partial` after activation, `failed` for stale/unavailable while bytes remain recovery-only, `superseded` or `burned` when a claim is lost, `retired` for a former active version after a newer publish, and `tombstoned` for the currently omitted version while its bytes remain recovery-only. If an interrupted upload cannot resume with the same digest and claim, the old `reserved` item is marked `burned` in the same CAS that reserves the replacement version. For replace-collection, any omitted document's reserved pending version is burned before its entry becomes `tombstoned`; the prior active ledger item is changed to `tombstoned` in that same CAS. A successful replacement changes the former active ledger item to `retired` in the same publish CAS, so only the current `document_attempt_id` can be `published_available` or `published_partial`. The active manifest tuple `(active_version, active_artifact_key, artifact_digest, backend, document_generation, document_attempt_id)` must match one ledger item and its `published_available`/`published_partial` state exactly. A stale or unavailable final transition marks its pending ledger item `failed` in the same CAS. `next_version` is maintained from this ledger under the same CAS and increases monotonically.

The artifact key is exact and versioned:

- `graphs/{collection}/{document_id}/v{version}/graph.json.gz`

The `v{version}` segment is the operation's reserved `pending_version`; the final artifact key never uses any other version choice.

Each versioned artifact key is write-once: create with `If-None-Match:*`; an existing object is idempotent success only when its canonical digest and metadata match exactly, otherwise the reservation is burned and the operation fails closed. Manifest CAS is the sole activation boundary; no writer overwrites or deletes immutable artifact bytes.

Replacement semantics are:

- `append` — reserve the pending manifest entry and `v1` under the manifest fence before any Qdrant or artifact work; reject duplicate appends for an already-tracked `document_id`; create generation `1` for a new document; publish only after validation succeeds.
- `replace-document` — reserve `pending_version=next_version` and the matching manifest claim under the manifest fence before any Qdrant or artifact work; record the new generation in the pending manifest entry, keep the prior pointer in the active fields for recovery while `pending` remains non-authoritative, then write the new artifact and publish `available` or `partial` only after validation succeeds.
- `replace-collection` — derive the incoming `document_id` set, apply the same document-level replacement rules only to included documents, and atomically write tombstoned entries for omitted documents in the active collection manifest by the same manifest CAS, incrementing `tombstone_epoch`. Omitted documents become non-discoverable tombstones: their immutable artifact bytes and recovery metadata are retained for audit and recovery, but they must not remain active or reader-discoverable; if the current entry is pending and still carries an active pointer, promote that exact pointer into `previous_pointer` before clearing the active fields, and if `previous_pointer` already exists keep the newest recoverable pointer. For replace-collection, write the durable Qdrant tombstone deny control point and completion proof before deletion, then retry deletion only after the tombstone is durable; deletion or point updates may touch only points that belong to the owning `pending_attempt_id` / `build_attempt_id`, generation, and collection fence token, or the operation must be serialized by the durable fence. Retrieval and compatibility fallback must filter points to `document_generation == manifest.document_generation` **and** `document_attempt_id == manifest.document_attempt_id`, exclude `graph_control_point` records, and exclude points/document_ids with `graph_tombstoned=true` plus tombstone epoch/generation/operation markers; the deny record is the source of truth for exclusion. If manifest reads/writes or Qdrant deny-record read/writes fail and the system cannot prove a document is not tombstoned, query must fail closed for that document or collection rather than using fallback. Missing or unavailable graph fallback remains only for documents proven not tombstoned. The manifest tombstone remains authoritative for normal state, and the Qdrant deny record/control point is the outage-safe safety net when the manifest is unavailable.

The first collection CAS sets `pending_tombstone_set_digest` to the literal `"pending"` marker and exposes the new epoch only as fail-closed state. After all retained and new control points are written/read-verified at that epoch, the writer computes the exact set digest and performs a second attempt-fenced CAS that sets `tombstone_set_digest` and clears `pending_tombstone_set_digest`; readers cannot proceed between those CAS operations.

Replace-collection publication order is two-phase and resumable: first claim the collection fence and CAS the tombstoned manifest entries with the next `tombstone_epoch`, `tombstone_attempt_id`, and `tombstone_fence_token`; then write/read-verify every Qdrant deny control point carrying that exact epoch and fence; only after all proofs are durable may deletion run. A crash or Qdrant failure leaves the manifest tombstoned and fail-closed, and the same claimed attempt resumes the missing proof. Reintroduction reverses the order: write and verify the new generation under a fresh document fence, clear the old deny point, then CAS the manifest to the new active state and epoch; a failed final CAS leaves the document denied and resumable.

Activation safety is:

- the manifest update must be atomic from the perspective of readers;
- stale writers must not overwrite a newer manifest pointer;
- replace/refresh must reserve the next version number under manifest CAS before any Qdrant work begins so the version cannot be reused by a different byte sequence; the reservation is `pending_version`, and the same operation may resume that reserved version only if it was interrupted without being superseded and the fencing fields still match;
- immediately before any publish or failure transition, reread the Qdrant `document_generation` and `source_fingerprint` for the same `document_id`, then perform one atomic manifest CAS/If-Match against the same `operation_id`, `document_generation`, `source_fingerprint`, `pending_operation_id`, `pending_generation`, `pending_source_fingerprint`, `pending_version`, `pending_attempt_id`, `build_attempt_id`, and current manifest ETag or fencing token; if that CAS fails, abort without changing status or ownership;
- the stale-transition CAS after a Qdrant mismatch must compare `operation_id`, `pending_attempt_id`, `build_attempt_id`, `document_generation`, `source_fingerprint`, `pending_version`, and the current manifest ETag or fencing token; if it fails, do nothing;
- replace-collection tombstones are written in the same manifest CAS that removes a document from the active set, and the tombstone retains immutable artifact bytes and recovery metadata while clearing the active pointer fields only after the newest recoverable pointer has been preserved in `previous_pointer`; write the Qdrant tombstone deny control point before deletion, retry deletion only after the tombstone is durable and read back cleanly, and fail closed if deletion still cannot complete or if the system cannot prove the document is not tombstoned;
- a stale operation cannot reintroduce a tombstoned entry because the resumer/finalizer must still match the current manifest fencing fields; any explicit future reintroduction uses a new replace-document generation and CAS, and append with the same `document_id` remains rejected;
- if the reread no longer matches the pending claim, reject publish, mark the entry `stale`, retain the prior pointer/bytes in the active fields for recovery, and keep the older pointer in `previous_pointer` only after a successful new publish has replaced it;
- if replacement fails after Qdrant advances, mark the entry `stale` or `unavailable`, set `failure_reason`, retain the prior pointer/bytes in the active fields for recovery, and never label the entry `available`; the prior bytes remain immutable and recoverable through the active fields and `previous_pointer` when a newer validated manifest is successfully published, but they are not reader-active;
- the older validated artifact bytes remain immutable and recoverable through `previous_pointer` until a newer validated manifest is successfully published, but they are not reader-active while the entry is pending, stale, or unavailable.

Raw relation evidence and the active graph are separate conceptual layers. Weak, rejected, or unverified candidates remain auditable without being allowed to dominate the active graph.


Existing collections can be backfilled from Qdrant payloads without re-embedding. Backfill ownership belongs to this ingest-stage lifecycle. Normal ingest uses deterministic operation ids of the form `ingest:{collection}:{document_id}:{document_generation}:{source_fingerprint}` and writes the pending manifest entry plus its version reservation before vector or artifact work starts. Backfill keeps its deterministic operation ids of the form `backfill:{collection}:{document_id}:{source_fingerprint}`. An entry is already complete only when it has terminal `available` status, a matching active pointer, an existing artifact, and matching metadata/digest. If an artifact upload was interrupted, resume only when the digest is an exact match, the same `pending_version` is still reserved, and the current manifest CAS fields plus the caller's inputs still authorize the pending claim; the current Qdrant metadata is then validation-only. Otherwise allocate `pending_version=next_version` and increment `next_version` under CAS. Existing manifests that do not yet have `next_version` initialize it under CAS to `max(existing_versions)+1`, and `next_version` remains monotonic across retries and supersessions. A superseded reservation is permanently burned; only an interrupted but not superseded matching operation may resume its exact `pending_version`. Missing, inconsistent, corrupt, or malformed payloads are unavailable/rebuild-required and are never guessed. `partial`, `pending`, `stale`, and `unavailable` resume or repair rather than complete. Reruns resume per document and preserve already active matching entries. Legacy points without `document_id` must be rebuilt or re-ingested because document ownership is never guessed. Legacy-point detection must scroll with offset until Qdrant returns `None`, validate every point, exclude any point with `graph_control_point` from legacy classification, and fail closed on any partial scan, missing page, or filter uncertainty.

Implementation ownership is fixed: PR A owns `ManifestStore`, scoped `FenceCoordinator`, immutable artifact writes, document/collection Qdrant control records, lifecycle activation, backfill with the baseline graph, reader tombstone preflight, manifest snapshot revalidation, and active `(document_generation, document_attempt_id)` retrieval filtering. PR D consumes that safety adapter and must not call unscoped Qdrant search directly. PR B may add bounded global relation recovery; PR C may add adaptive topology/community selection; PR D may add query-time context allocation. PR B-D may extend artifact payloads, but none may redefine manifest CAS, fence scopes, tombstone predicates, digest validation, version-ledger states, or fallback deny semantics. Concrete mapping: PR B covers issues #45–#48, PR C covers #49–#51, and PR D covers #40 and #52; each consumes PR A safety interfaces rather than reimplementing them.

## Alternatives rejected

- **Build the full graph after every query:** repeats expensive work and prevents reusable cross-chunk document relations.
- **Persist the first collection-global graph:** creates cross-document identity collisions and complicates incremental append/replace operations.
- **Use Neo4j as a required graph store:** adds operational and dependency cost before graph scale requires it.
- **Add Parquet/`pyarrow` immediately:** adds a dependency and format decision before artifact size is measured; compressed JSON is sufficient for the first version.
- **Fail vector ingest when graph construction fails:** reduces data availability and makes optional LLM/storage failures destructive to the ingest path.
- **Bundle adaptive topology or query-time context allocation into this ADR:** those decisions are owned by stacked follow-on PRs #64/#65 and planning issues #60/#61, not this document.

## Consequences

Positive:

- Graph artifacts become versioned, inspectable, and rebuildable.
- Ingest owns artifact creation, validation, backfill, manifest activation, and replacement safety.
- Existing Qdrant and R2/MinIO infrastructure remains the storage contract.
- Query-Only and downstream behavior remain unchanged while the ingest-stage graph artifact stays optional.

Costs and constraints:

- Ingest becomes more expensive and may require bounded entity and relation processing.
- Graph artifacts need versioning, manifest lifecycle, status reporting, stale-write protection, and backfill tooling.
- Query-time code must handle missing, stale, or tombstoned graph artifacts safely; only a document proven not denied by the mandatory preflight may use compatibility fallback, `partial` artifacts remain diagnostic-only, and tombstoned entries are skipped as non-discoverable.
- A collection spanning multiple documents still requires temporary in-memory graph composition for cross-document queries; no persistent cross-document graph is introduced here.
