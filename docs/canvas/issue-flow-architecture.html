Flow architecture: detailed impact

Issue work is not a separate product; each wave changes one contract in the same Full-Pipeline seam.

1. Ingest and Collection Target

Before: each PDF could restart chunk_id from zero, and a large hierarchy-aware upload could exceed Qdrant Cloud request limits. A shared collection was safe only by convention.

After: #34 writes bounded batches. #12 adds document_id, chunk_uid, deterministic UUID point IDs, and explicit append, replace-document, and replace-collection semantics. Legacy points without document metadata fail closed for document-scoped operations.

Architecture effect: retrieval can now return mixed-document chunks without local-ID collision, which gives graph and pruner stages a stable identity seam.

2. Retrieval and evidence preparation

Delivered baseline: #26 preserves hierarchy/layout metadata, #27 adds path-aware selection, and #37–#38 now deliver bounded parent context and tiny-sentence filtering.

Open direction: #36 would make tables and figures first-class evidence. #40 adds a richer optional path signal.

Architecture effect: selection moves from “top similar chunks” to structure-aware evidence with explicit reasons, without changing Query-Only or Ingest launcher modes.

3. Local graph → global graph recovery

Before: entity/relation extraction is mostly local to a chunk, and fallback co-occurrence can create dense edges without recording why they exist.

After target: #45 records evidence type and edge weight; #46 canonicalizes conservative entity variants and classifies weak/orphan regions; #47 creates bounded cross-chunk candidates; #48 verifies candidates and cleans unsupported noise while protecting query-relevant chunks.

Architecture effect: the graph becomes a recoverable evidence system rather than an opaque adjacency structure. #39 is the reliability seam consumed by #48.

4. Graph topology → communities

Before: fixed kNN/threshold edges feed one Leiden modularity partition.

After target: #49 keeps the fixed policy as baseline but adds bounded adaptive topology. #50 explores bounded Leiden candidates and selects a partition using normalized graph/semantic signals. #51 compares the active partition with agglomerative chunk clustering as a diagnostic only.

Architecture effect: community boundaries become reproducible, inspectable decisions instead of a single unchallenged library output.

5. Communities → final context and summary

Before: fixed per-community/global top-k can overfeed repetitive communities and underfeed evidence-rich ones.

After target: #52 allocates one bounded budget by community importance, then selects chunks using relevance, graph/relation support, section diversity, and novelty. It records inclusion/rejection reasons and keeps query-protected chunks eligible.

Architecture effect: the final summarizer receives less redundant, more explainable context while existing NAP/CAP/CGM prompts, RAPTOR reduction, evaluation, and bounded feedback remain downstream contracts.

Architectural decisions that keep the system small

What the issue program deliberately does not introduce.

Qdrant remains the vector seam

Document-safe point IDs and payload metadata are enough for #12. No Neo4j is required for the current flow.

No Microsoft GraphRAG import

#44 builds bounded local contracts and artifacts on existing NetworkX/Leiden/embedding seams; it does not reproduce a second framework.

Artifact-first observability

Adaptive decisions should write attempt-scoped JSON artifacts: evidence, candidates, verification, topology, communities, and context budget.

Persistent graph-at-ingest is next delivery

ADR 0002 is the decision record for persistent graph-at-ingest. PR #62 still needs a separate implementation delivery issue for PR A, and the R2 manifest/artifact contract is planned delivery rather than a standalone unplanned issue.

Practical order

Review and merge the active docs stack #62 → #63 → #64 → #65, plus #66, into main. Then create the separate PR A delivery issue for persistent ingest graph work, followed by bounded frontiers #45 and #49. Do not wait for #41 before #51: both reuse embeddings, but their pipeline stages are different.
